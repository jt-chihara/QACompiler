# PRD: TODO App — 期限・優先度機能の追加

## 背景

現在のTODOアプリはタイトルと完了状態のみを管理するシンプルなCRUDアプリケーションである。ユーザーからのフィードバックにより、以下の機能が強く求められている:

- **期限 (Due Date)**: いつまでにやるべきかを設定したい
- **優先度 (Priority)**: どのTODOを先にやるべきか分かるようにしたい

## ゴール

TODOに期限と優先度を追加し、ユーザーが「何をいつまでにやるべきか」を一目で把握できるようにする。

## 機能要件

### FR-1: 優先度の設定

- TODOに `high` / `medium` / `low` の3段階の優先度を設定できる
- デフォルトは `medium`
- 一覧画面で優先度に応じた視覚的な区別を行う（色分け）
  - `high`: 赤系 (`#e53e3e`)
  - `medium`: 黄系 (`#d69e2e`)
  - `low`: 灰系 (`#a0aec0`)
- 作成後も優先度を変更できる

### FR-2: 期限の設定

- TODOに任意で期限日（日付のみ、時刻なし）を設定できる
- 期限なしのTODOも許容する
- 期限切れのTODOは視覚的に警告表示する（赤字 + アイコン）
- 期限が当日のTODOは「今日まで」と表示する
- 完了済みTODOでは期限切れ警告を表示しない

### FR-3: ソート

- 一覧画面でTODOをソートできる
  - ソート基準: 優先度（高→低）、期限（近い順）、作成日（新しい順）
- デフォルトソート: 優先度降順 → 期限昇順

### FR-4: フィルタリング

- 優先度でフィルタリングできる（複数選択可）
- 「期限切れのみ」フィルタを用意する
- 完了済みの表示/非表示を切り替えられる

## 非機能要件

### NFR-1: パフォーマンス

- TODO 1,000件でも一覧表示が200ms以内
- ソート・フィルタ操作がクライアントサイドで即座に反映される

### NFR-2: データ整合性

- 期限は `YYYY-MM-DD` 形式（ISO 8601の日付部分）
- 過去の日付も期限として設定可能（入力時にバリデーションエラーにしない）
- 優先度は必ず3値のいずれか（不正な値はAPIで拒否）

### NFR-3: 後方互換性

- 既存のTODO（優先度・期限なし）は引き続き動作する
- 既存APIのレスポンス構造に追加フィールドとして含める（破壊的変更なし）

## 対象外

- 繰り返しタスク
- リマインダー通知（メール、プッシュ通知）
- カレンダービュー
- 複数ユーザー対応
